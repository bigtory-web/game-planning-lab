# 게임 기획용 CLAUDE.md 가이드

## 이 문서는 무엇인가

이 문서는 게임 기획 작업에 맞춰 커스터마이징한 `CLAUDE.md`의 의미를 한국어로 풀어쓴 가이드입니다.

목적은 3가지입니다.

1. 왜 저장소 이름을 `game-planning-lab`으로 추천했는지 정리
2. 추가한 `CLAUDE.md` 항목을 한글로 해석
3. 각 항목이 왜 필요한지, 어떤 의도로 넣었는지 설명

---

## 왜 `game-planning-lab`인가

`spin-miner-prototype` 같은 이름은 특정 프로젝트 하나에는 잘 맞지만, 이후에 다른 기획서, 다른 프로토타입, 공용 프롬프트까지 같이 쌓이기 시작하면 이름과 실제 용도가 어긋나기 쉽습니다.

`game-planning-lab`을 추천한 이유는 아래와 같습니다.

### 1. 프로젝트 하나가 아니라 작업 허브에 가깝기 때문

당신의 작업은 보통 아래 3개가 같이 움직입니다.

- 기획서
- 프로토타입
- 프롬프트

그리고 이 셋은 서로 연결됩니다.

- 특정 기획서에 대응하는 프로토타입이 있음
- 그 프로토타입을 만들기 위한 프롬프트가 있음
- 공용으로 재사용할 프롬프트도 따로 있음

그래서 저장소 이름도 특정 게임명이 아니라, 더 넓은 개념이 맞습니다.

### 2. 프로젝트가 계속 늘어나도 어색하지 않음

예를 들어 아래처럼 확장해도 자연스럽습니다.

- `projects/spin-miner/`
- `projects/mercenary-reselect/`
- `prompts/common/`
- `templates/`

### 3. 포트폴리오 작업소라는 느낌이 있음

`lab`이라는 말은 실험, 정리, 검증, 반복 개선의 느낌이 있어서 기획서와 프로토타입을 함께 운영하는 작업 방식과 잘 맞습니다.

---

## 추천 폴더 구조

```text
game-planning-lab/
  CLAUDE.md

  projects/
    mercenary-reselect/
      docs/
      prototype/
      prompts/
      assets/

    spin-miner/
      docs/
      prototype/
      prompts/
      assets/

  prompts/
    common/

  templates/

  tasks/
    todo.md
    lessons.md

  archive/
```

### 구조 의도

- `projects/`: 실제 진행 중인 프로젝트별 작업물
- `docs/`: 기획 문서, 메모, 정리 글
- `prototype/`: 실행 가능한 프로토타입
- `prompts/`: 프로젝트 전용 프롬프트
- `prompts/common/`: 여러 프로젝트에서 돌려 쓸 공용 프롬프트
- `templates/`: 새 문서나 새 프로토타입 시작용 템플릿
- `tasks/todo.md`: 현재 할 일 관리
- `tasks/lessons.md`: 반복 실수와 교정 규칙 기록
- `archive/`: 지금은 안 쓰지만 참고 가치가 있는 자료 보관

---

## `archive/`에 무엇을 넣는가

`archive/`는 단순히 오래된 파일을 버리는 폴더가 아니라, 현재는 비활성이지만 나중에 참고할 수 있는 자료를 모아두는 곳입니다.

예를 들면:

- 떨어진 회사 맞춤 문서
- 방향 바꾸기 전 초안
- 폐기된 시스템 구조안
- 더 이상 안 쓰는 프로토타입 버전
- 예전 프롬프트 버전

### 의도

- 현재 진행 방향과 과거 자료를 분리
- 활성 작업 폴더를 깔끔하게 유지
- 그래도 나중에 아이디어 회고나 포트폴리오 정리에 다시 참고 가능하게 유지

## 추가한 게임 기획용 `CLAUDE.md` 한글 해석 + 의도

아래는 내가 제안한 `CLAUDE.md`를 섹션별로 해석하고, 왜 넣었는지 설명한 것입니다.

---

## 1. Role

### 원문 의미

클로드를 단순 설명 도구가 아니라, 아이디어를 포트폴리오용 시스템, 문서, 프로토타입으로 다듬는 게임 기획 파트너로 정의하는 부분입니다.

### 왜 넣었나

일반 AI는 쉽게 “설명 잘하는 도우미”처럼 흘러갑니다.  
하지만 당신이 원하는 건:

- 시스템 기획 관점
- 포트폴리오 가치 판단
- 논리 검토
- 프로토타입 연결

이므로, 역할부터 명확히 고정해주는 게 중요합니다.

---

## 2. Working Principles

### 2-1. Plan Before Expansion

#### 원문 의미

복잡한 작업은 바로 확장하지 말고 아래 순서로 정리하라는 뜻입니다.

1. 의도
2. 문제 정의
3. 핵심 규칙
4. 리스크
5. 대응
6. 남는 과제
7. 프로토타입/문서 방향

#### 왜 넣었나

당신과의 대화에서 자주 나온 문제가:

- 아이디어가 커지면서 중심이 흐려짐
- 장치가 늘어나는데 정체성은 약해짐

이었기 때문입니다.

즉, 무조건 “더 생각”이 아니라 “구조를 먼저 세우기”가 필요해서 넣은 규칙입니다.

### 2-2. Be Concise

#### 원문 의미

길게 반복 설명하지 말고, 이미 성립한 딜레마는 또 반복하지 말라는 뜻입니다.

#### 왜 넣었나

당신이 여러 번:

- 말이 너무 길다
- 도돌이표 같다
- 이미 상중하로 파악했다

라고 했기 때문에, 이건 반드시 명시해야 하는 작업 스타일이었습니다.

### 2-3. Think Like a System Planner

#### 원문 의미

일반적인 도움말이 아니라, 모바일 게임 시스템 기획자 시점으로 보라는 뜻입니다.

항상 봐야 할 것:

- 무엇을 해결하는가
- 시스템 정체성이 한 줄로 보이는가
- 유저가 이해할 수 있는가
- 라이브 서비스에서 버틸 수 있는가
- 포트폴리오로 가치가 있는가

#### 왜 넣었나

당신은 “설명”보다 “기획 판단”을 원합니다.  
그래서 모든 답변이 시스템 검토처럼 나와야 한다는 기준을 넣었습니다.

### 2-4. Detect Structural Failure Early

#### 원문 의미

좋게 포장하기 전에 먼저 구조적 결함을 보라는 뜻입니다.

특히:

- 무한 루프
- 결과물이 다시 입력 자산이 되는지
- 어뷰징 가능성
- 시간/재화로 원하는 결과에 쉽게 수렴하는지
- 기존 획득/성장 의미를 무너뜨리는지

를 먼저 확인하라는 규칙입니다.

#### 왜 넣었나

용병 전환 시스템 논의에서 가장 중요했던 순간이 바로 이 부분이었습니다.

예:

- 전환 결과가 다시 전환 재료가 되면 무한 루프
- 직접 획득보다 전환이 더 기대되면 구조 흔들림

이걸 초기에 못 보면 시스템이 계속 헛도는 걸 경험했기 때문에 넣었습니다.

### 2-5. Preserve Core Identity

#### 원문 의미

고치다가 아예 다른 시스템이 되어버리면 그걸 분명히 말하라는 뜻입니다.

#### 왜 넣었나

당신은 여러 번:

- 이건 세팅형으로 가면 그냥 장착 시스템 아냐?
- 이건 아예 다른 구조로 바뀌는 거 아냐?

를 정확히 짚었습니다.

즉 수정안이 “개선”인지 “변질”인지 구분하는 게 중요해서 넣었습니다.

### 2-6. Distinguish Main vs Supporting System

#### 원문 의미

아이디어가 메인 포트폴리오 시스템인지, 보조 시스템인지, 단순 조정 장치인지 구분하라는 뜻입니다.

#### 왜 넣었나

당신은 단순히 “좋은 시스템”보다:

- 메인 포폴로 밀 가치가 있나
- 서브 시스템으로 더 맞나

를 중요하게 봅니다.

실제로 용병 전환 시스템도 “좋지만 메인보단 서브에 가깝다”는 판단이 있었기 때문에 이 규칙을 넣었습니다.

---

## 3. Review Format

### 원문 의미

시스템 검토를 아래 구조로 하라는 뜻입니다.

- intent
- problem
- structure
- risk
- response
- remaining issue

필요하면 `상/중/하` 같은 위험도를 쓰되, 기계적으로 남발하지 말라는 뜻도 포함됩니다.

### 왜 넣었나

당신이 좋아한 정리 방식이 딱 이 구조였기 때문입니다.

예:

- 무한 전환 가능성(상)
- 검토 포인트
- 대응
- 남는 과제

즉 대화 내용을 문서형 검토로 바로 옮길 수 있는 구조를 고정한 것입니다.

---

## 4. Prototype Rules

### 4-1. Prototype Goal First

#### 원문 의미

프로토타입 만들기 전에, 뭘 검증하려는 프로토타입인지 먼저 정하라는 뜻입니다.

예:

- 규칙 검증
- UX 검증
- 확률 체감 검증
- 감정 반응 검증

#### 왜 넣었나

프로토타입은 아무리 예뻐도 목적이 없으면 흔들립니다.  
당신은 특히 “체감 검증용”과 “문서 보조용”을 구분할 필요가 있어서 넣었습니다.

### 4-2. Prototype Scope

#### 원문 의미

프로토타입을 너무 크게 만들지 말고, 테스트 가능한 작은 단위로 만들라는 뜻입니다.

#### 왜 넣었나

당신은 비개발자이기 때문에, 모든 프로토타입을 완성 게임처럼 만들 필요가 없습니다.  
핵심 흐름만 검증하는 쪽이 더 효율적이라 넣었습니다.

### 4-3. UX Matters

#### 원문 의미

게임 시스템 프로토타입에서는 UI가 단순 껍데기가 아니라 시스템 일부라고 보라는 뜻입니다.

봐야 할 것:

- 유저가 처음 뭘 보는가
- 어떤 결정을 하는가
- 어떤 피드백을 받는가
- 감정적으로 흐름이 이해되는가

#### 왜 넣었나

용병 재선택 시스템도 결국:

- 원본 카드
- 전환 버튼
- 후보 리스트
- 최종 선택

이 흐름이 체감의 핵심이었기 때문입니다.

### 4-4. Document-Prototype Link

#### 원문 의미

프로토타입이 문서 기반이면, 무엇을 단순화했고 무엇을 뺐는지 분명히 하라는 뜻입니다.

#### 왜 넣었나

포트폴리오에서는 기획 문서와 프로토타입이 서로 어긋나면 신뢰가 떨어집니다.  
그래서 둘을 연결하는 규칙을 넣었습니다.

---

## 5. Portfolio Rules

### 5-1. Always Test Portfolio Strength

#### 원문 의미

아이디어를 볼 때 항상 포트폴리오 강도를 같이 보라는 뜻입니다.

질문 예시:

- 한 줄 특징이 보이는가
- 플레이 체감을 바꾸는가
- 너무 흔한가
- 메인으로 약한가
- 서브 시스템에 더 맞는가

#### 왜 넣었나

당신은 “좋은 시스템인가?”보다  
“포트폴리오로 낼 만큼 선명한가?”를 더 자주 묻습니다.  
그래서 별도 섹션으로 넣었습니다.

### 5-2. Favor Honest Evaluation

#### 원문 의미

애매한 시스템을 괜히 좋다고 포장하지 말라는 뜻입니다.

#### 왜 넣었나

당신은 무난한 아이디어를 싫어하고,  
“그냥 평범하면 평범하다고 말해달라”는 기대가 강합니다.

### 5-3. Avoid Empty Originality

#### 원문 의미

억지로 신기한 것만 찾지 말고, 익숙한 시스템이어도 문제 정의와 논리가 강하면 괜찮다는 뜻입니다.

#### 왜 넣었나

당신은 “다른 회사도 이미 아는 시스템 아니야?”를 계속 검토했지만, 동시에 완전히 뜬구름 잡는 시스템도 원하지 않았습니다.  
이 균형을 잡기 위해 넣었습니다.

---

## 6. Prompt Support Rules

### 원문 의미

프롬프트를 만들 때도 일반용이 아니라, 게임 기획 목적에 맞춘 전용 프롬프트를 만들라는 뜻입니다.

예:

- 시스템 아이디어 발상용
- 어뷰징 검토용
- 포트폴리오 문서용
- 프로토타입 개발용

### 왜 넣었나

당신은 “그냥 설명 프롬프트”가 아니라  
“게임 기획자 프롬프트”를 원했습니다.  
그래서 이 부분은 꼭 있어야 했습니다.

---

## 7. Project Organization Rules

### 원문 의미

저장소와 파일을 어떻게 운영할지에 대한 규칙입니다.

핵심:

- `projects/`에 활성 프로젝트
- `prompts/common/`에 공용 프롬프트
- `templates/`에 템플릿
- `tasks/todo.md`에 현재 작업 목록
- `tasks/lessons.md`에 반복 실수 기록
- `archive/`에 비활성 자료 보관

### 왜 넣었나

당신이 물었던 핵심이:

- 새 저장소를 어떻게 구성할지
- 기획서, 프로토타입, 프롬프트를 어떻게 연결할지
- 공용 프롬프트와 프로젝트 전용 프롬프트를 어떻게 나눌지

였기 때문입니다.

---

## 8. My Preferences

### 원문 의미

당신이 선호하는 답변 스타일과 작업 성향을 명시한 부분입니다.

예:

- 짧고 직접적인 답
- 반복 경고 싫음
- 논리적 비판 선호
- 현실 서비스 기준 중요

### 왜 넣었나

이 부분이 없으면 AI는 다시 일반적인 길고 중립적인 톤으로 돌아갈 가능성이 큽니다.

---

## 9. If I Correct You

### 원문 의미

당신이 논리 오류, 어뷰징 포인트, 반복 설명 같은 걸 지적하면  
사과만 하지 말고 바로 추론을 수정하고 앞으로 반영하라는 뜻입니다.

### 왜 넣었나

실제로 대화 중에:

- 무한 전환 루프
- 메타 수렴보다 더 중요한 구조적 문제
- 반복되는 설명 방식

같은 걸 당신이 먼저 잡아냈습니다.  
그래서 이건 꼭 강하게 넣어야 하는 규칙이었습니다.

## 이 문서를 어떻게 쓰면 좋은가

추천 순서는 이렇습니다.

1. 이 가이드를 읽고 마음에 드는 규칙만 먼저 고르기
2. 그 규칙들만 `CLAUDE.md`에 넣기
3. 실제로 몇 번 같이 작업해보기
4. 안 맞는 문장이나 과한 부분은 지우기
5. 반복적으로 필요한 규칙만 남기기

즉, 처음부터 완벽하게 만들려고 하기보다  
**쓰면서 다듬는 작업 규칙**으로 보면 됩니다.

## 한 줄 정리

이 `CLAUDE.md`는 단순한 프롬프트가 아니라,

**게임 기획 작업에서**
- 아이디어를 어떻게 검토할지
- 프로토타입을 어떻게 만들지
- 포트폴리오 강도를 어떻게 판단할지
- 작업 파일을 어떻게 운영할지

를 미리 고정해두는 운영 문서입니다.
